# triage

A Rust TUI to **triage parallel Claude Code sessions across tmux panes** ‚Äî sort by attention priority so you know which session needs you next.

Joins three signal sources Claude Code already writes:

- `~/.claude/sessions/<pid>.json` ‚Äî discovery + `idle`/`busy` status
- transcript JSONL at `~/.claude/projects/<encoded-cwd>/<sessionId>.jsonl` ‚Äî pulls Claude's natural-language **`away_summary`** recap, `last-prompt`, `turn_duration`
- `tmux list-panes -a` ‚Äî joined via process-ancestor walk from the Claude PID

into one sortable, filterable table. Jump to any session's tmux pane with `‚èé`.

## Why

If you run multiple Claude Code sessions in tmux on different topics, you can't tell at a glance which needs attention. `triage` reads the recap Claude writes every ~3 minutes idle, surfaces it inline per row, and sorts by attention need ‚Äî so you see which session just finished, which is still working, which is long-idle.

Different shape from the existing [`agentop`](https://crates.io/crates/agentop) (which is process-centric, token/cost-focused). `triage` is content-centric: the headline column is the recap, not a status badge.

## Usage

```bash
cargo install --path .
triage              # launch the TUI
triage --probe      # print the joined session table once (no TUI) ‚Äî useful for scripting
```

### Keys

| Key | Action |
| --- | --- |
| `‚Üë` `‚Üì` / `j` `k` | move selection |
| `‚èé` | jump to selected session's tmux pane |
| `space` | toggle detail panel |
| `/` | filter (substring across name/cwd/headline/pane) |
| `r` | request manual refresh |
| `q` / `Ctrl-C` | quit |

### Attention states (default sort order)

| State | Meaning |
| --- | --- |
| `error` | Last `stop_hook_summary` reported errors. |
| `done` | `Stop` observed within last 3 min ‚Äî likely awaiting next prompt. |
| `work` | sessions JSON `status = busy`. |
| `fresh` | No user prompts seen yet. |
| `idle` | `Stop` observed >3 min and <30 min ago. |
| `long` | `Stop` observed >30 min ago. |
| `?` | Indeterminate. |

## Design notes

- **Zero hooks.** Reads only files Claude Code already writes. No `~/.claude/settings.json` edits required.
- **Mechanical signal extraction.** No LLM in the live path; `away_summary` is the recap, generated by Claude itself every ~3 min idle.
- **Discovery + tmux join.** Sessions JSON is keyed by the Claude PID. Tmux's `pane_pid` is the shell; we walk the process tree upward from the Claude PID until we hit a `pane_pid`.
- **Transcript join via sessionId, fallback to newest mtime.** Sessions JSON `sessionId` can lag after `/clear`; mtime fallback handles that case.
